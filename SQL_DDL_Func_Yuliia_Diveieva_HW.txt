Task 7: What does ‘last_updated’ function do? Where is it used?
The function automatically updates a timestamp column to the current time whenever a row in a table is modified. It is used in the most of the tables of the DB.
The function is used in different scenarios. The main one is that when we have to move data from the current production database to the warehouse one it is better to do by comparing the timestamp of the last download with the last_updated timestamp instead of a full load, that significantly reduces processing time and resource consumption.
Also it is really useful when the bag is discovered to find the point in time when it happened. This allows for better root-cause analysis and helps identify other records that might have been affected during the same timeframe.

Task 9: Can any function produce incorrect results? Why?
There are many factors that can cause incorrect results even when the function syntax is perfectly valid. First of all, the underlying logic may be broken, leading the query to produce wrong results, or sometimes no results at all, despite appearing to work. For example, a function like inventory_in_stock() might only look for a "return date" in the rental table to decide if a movie is available. If a DVD is damaged or stolen, that return date remains empty forever, causing the function to incorrectly report the item as "rented" instead of "decommissioned," leading to ghost inventory that doesn't actually exist on the shelf.
Not including logic to counteract NULL values also frequently leads to errors, as many functions behave unexpectedly when these values are present in a dataset. Consider the get_customer_balance() function: if it attempts to sum up payments but encounters a NULL value in a fee column without using a helper like COALESCE, the entire mathematical operation might return NULL or an incomplete total. This results in the database claiming a customer owes nothing simply because one piece of data was missing, which creates significant financial discrepancies.
Furthermore, it is dangerous to hardcode IDs within functions. These IDs contain information specific only to one particular database instance and are incredibly hard to control when transferring data or correcting mistakes. For instance, if you hardcode category_id = 5 because it represents "Comedy" in your local setup, the function will break or return the wrong genre if you move the code to a production server where "Comedy" is assigned id = 10. This makes the code fragile and difficult to maintain as the project grows.
