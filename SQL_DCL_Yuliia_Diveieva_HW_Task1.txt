1.
To find all superusers, I used the following query:
SELECT rolname, rolsuper, rolcanlogin, rolconnlimit
FROM pg_roles
WHERE rolname NOT LIKE 'pg_%';

It showed that the DB has only one user "postgres" which has full access. Now, let’s look at other existing roles:

SELECT 
    rolname AS role_name, 
    rolsuper AS is_superuser, 
    rolcreaterole AS can_create_roles, 
    rolcreatedb AS can_create_db, 
    rolcanlogin AS can_login, 
    rolreplication AS is_replication_role,
    rolconnlimit AS connection_limit
FROM pg_roles;

By running this query, we see all roles and the types of privileges they possess. It confirms that no other role has any privileges except for the "postgres" superuser.

2.
SELECT grantee, table_name, privilege_type 
FROM information_schema.table_privileges;

Running this query shows that the "postgres" role is the only role with permissions, as expected.

3.
SELECT tablename, rowsecurity

There is no row-level security enabled in the DB.

4.
SELECT name, setting, short_desc
FROM pg_settings
WHERE name IN ('password_encryption', 'max_connections', 'ssl');

The max_connections limit is 100, password_encryption is set to scram-sha-256, and ssl is off.
While having SSL disabled is acceptable for a learning database, in a production environment, it could expose data to unauthorized users. It would be better to switch it to a secure mode. Similarly, a max_connections limit of 100 could significantly restrict operations in a real world if many users attempt to connect at the same time.

The database security is currently centered entirely around the "postgres" user. Importantly, no permissions have been granted to the "public" role, which a good sign of high security.
